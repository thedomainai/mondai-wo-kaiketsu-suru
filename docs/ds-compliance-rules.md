# デザインシステム準拠ルール — エージェント向けガイドライン

## このドキュメントの目的

スライドデッキ（1280x720px 固定寸法）のデザインシステム（DS）準拠チェックを行うエージェントが、**何を自動修正してよいか（HARD）** と **何をレポートのみに留めるべきか（SOFT）** を明確に区分するためのルールである。

### 背景：なぜ区分が必要か

過去に 81 エージェントがスライドを並列レビューし、DS 違反を機械的に修正した結果、広範なレイアウト崩壊が発生した。原因は以下の通り。

- **非標準スペーシングトークン**（`--sp-14`, `--sp-18`, `--sp-22` 等）を最寄りの標準トークンに丸めた結果、累積的なピクセルずれが発生
- **margin を gap に変換** したことでレイアウトのセマンティクスが変わった
- **装飾ボーダーの除去** が視覚的なグルーピングを破壊した
- **インラインスタイルの CSS クラス化** により、スライド固有の微調整が失われた

**これらの修正はすべて DS 準拠の観点では「正しい」が、視覚的には「壊した」。** この区分はその教訓から生まれたものである。

## 大前提：固定寸法スライドデッキの制約

### Web アプリの DS トークンスケールがそのまま適用できない理由

このスライドデッキは **1280x720px の固定キャンバス** 上で表示される。Web アプリケーションとは以下の点で根本的に異なる。

| 特性 | Web アプリ | 固定寸法スライド |
|---|---|---|
| ビューポート | 可変（レスポンシブ） | 固定（1280x720px） |
| コンテンツ量 | スクロールで収容 | 1 画面に収めなければならない |
| ピクセル精度 | 数 px のずれは許容 | 1px のずれが視認できる |
| レイアウト手法 | フロー + グリッド | 絶対配置 + ステージ内の flex/grid |
| DS トークンの役割 | 一貫性の担保（主目的） | 一貫性 + ピクセル精度（両立が必要） |

**したがって、以下のことが起こる。**

1. **スペーシング 8 段スケール（4, 8, 12, 16, 20, 24, 32, 48）では足りない場面がある。** たとえば `--stage-top: 184px` はヘッダーコピーの高さに応じて決まるスライド固有の座標であり、スケールに載せる対象ではない。

2. **図形やツリーの絶対座標は DS トークン化の対象外。** SVG の `cx="150"` や CSS の `left: 48px` は図形の幾何学であり、スペーシングスケールとは無関係。

3. **`1px` の gap や `6px` の gap は意図的な値。** グリッドのセパレータ線（`gap:1px;background:var(--ink-08)`）やアイコンの微小な隙間は、スペーシングスケール外であっても正当。

## HARD 違反（自動修正可）

以下の項目は**視覚的なレイアウトに影響しない構造的問題**であり、エージェントが発見次第、自動修正してよい。

### H01: `<style>` ブロックの存在

**定義:** スライド HTML の `<head>` または `<body>` 内に `<style>` タグが存在する。

**現状:** 10 スライドに `<style>` ブロックが存在（slide_18, slide_21, slide_24, slide_34, slide_35, slide_41, slide_45, slide_46, slide_64）。

**なぜ HARD か:** `<style>` ブロックの存在自体はレイアウトを変えない。slides.css に同等のクラスがあればそれに移行し、なければ slides.css にクラスを追加してからインラインスタイルまたはクラス参照に置き換える。

**エージェントがすべきこと:**
1. `<style>` ブロック内の CSS ルールを読む
2. slides.css に同等のクラスが存在するか確認する
3. 存在する場合 → HTML 要素のクラス属性を更新し、`<style>` ブロックを削除する
4. 存在しない場合 → slides.css にスライド固有のクラスを追加し（例: `.mece-venn`, `.mece-circle`）、HTML をクラス参照に書き換え、`<style>` ブロックを削除する
5. **修正前後のブラウザ表示が変わらないことを確認する**

**エージェントがしてはならないこと:**
- CSS プロパティの値を「DS 準拠」に変更しながら `<style>` を移動すること（2 つの変更を同時にやらない。まず構造を移し、値の変更は別ステップで検討する）

**具体例:**
```html
<!-- NG: slide_46.html -->
<style>
  .mece-venn { position: relative; width: 178px; height: 160px; }
  .mece-circle { position: absolute; top: 24px; ... }
</style>

<!-- OK: slides.css に移動 -->
<!-- slides.css に .mece-venn, .mece-circle を追加し、HTML から <style> を削除 -->
```

### H02: Google Fonts リンクの欠落

**定義:** `<head>` 内に Google Fonts の `<link>` タグがない、またはフォント指定が不完全（Space Grotesk / Noto Sans JP のいずれかが欠落）。

**なぜ HARD か:** フォントリンクがなくても CSS は適用される（fallback フォントで表示される）が、デザインが崩れる。リンクの追加はレイアウト座標に一切影響しない。

**エージェントがすべきこと:**
```html
<!-- 以下のリンクを <head> 内に追加する（既存と同一であること） -->
<link href="https://fonts.googleapis.com/css2?family=Noto+Sans+JP:wght@400;700;900&family=Space+Grotesk:wght@700;900&display=swap" rel="stylesheet"/>
```

**エージェントがしてはならないこと:**
- ウェイト指定を勝手に変更すること（`wght@400;700;900` は Noto Sans JP の仕様。Space Grotesk は `wght@700;900`）
- プリコネクトヒントなど最適化を勝手に追加すること

---

### H03: slides.css または slides.js リンクの欠落

**定義:** `<head>` 内に `<link href="./slides.css">` または `<script defer src="./slides.js">` が存在しない。

**なぜ HARD か:** リンクの追加はレイアウトの意図を変えない。ただし slides.css のリセットルール（`p { margin: 0 }` 等）が適用されるため、**追加後にレイアウトが変わる可能性がある**。そのため、追加後は必ず表示確認する。

**エージェントがすべきこと:**
```html
<link href="./slides.css" rel="stylesheet"/>
<script defer src="./slides.js"></script>
```

**エージェントがしてはならないこと:**
- slides.css の追加と同時に他の修正を行うこと（CSS リセットの影響を切り分けるため、1 ステップで完結させる）

---

### H04: data-header-* / data-footer 属性の欠落

**定義:** コンテンツスライド（`data-slide-kind="content"`）に `data-header-chapter`, `data-header-title`, `data-footer="standard"`, `data-footer-page` のいずれかが欠落している。

**なぜ HARD か:** これらの属性は slides.js がヘッダー/フッターを動的生成するためのデータ属性であり、追加してもコンテンツ領域のレイアウトには影響しない。

**エージェントがすべきこと:**
1. デッキの章構成（docs/slide-outline.md 等）からスライドが属する章を特定する
2. 正しい `data-header-chapter`, `data-header-title`, `data-header-subtitle` を設定する
3. `data-footer="standard"` と `data-footer-page="NN"`（スライド番号）を設定する

**エージェントがしてはならないこと:**
- 章扉スライド（`data-slide-kind="divider"`）やタイトルスライド（`data-slide-kind="cover"`）に標準ヘッダー属性を付けること（これらは独自レイアウト）

**ページ番号の注意:**
- `data-footer-page` はスライドファイル名の番号（slide_07.html → `"07"`, slide_46.html → `"46"`）と一致させる
- 過去に off-by-one バグが 2 回発生している。ファイル名との一致を必ず確認すること

---

### H05: 存在しないトークンの参照

**定義:** CSS 内で定義されていないカスタムプロパティを参照している。

**なぜ HARD か:** 存在しないトークンは CSS の `var()` フォールバックが効かず、プロパティがデフォルト値にリセットされる。意図と異なる表示になる。

**slides.css に定義されているトークン一覧:**

カラー:
```
--ink-100, --ink-82, --ink-68, --ink-50, --ink-34, --ink-18, --ink-08, --ink-04
--surface-white, --surface-mist, --surface-card
--accent-purple, --accent-red
--soft-black (= var(--ink-100) のエイリアス)
```

スペーシング:
```
--sp-4, --sp-8, --sp-12, --sp-16, --sp-20, --sp-24, --sp-32, --sp-48
```

フォントサイズ:
```
--fs-11, --fs-12, --fs-13, --fs-14, --fs-15, --fs-16, --fs-17, --fs-18
--fs-22, --fs-24, --fs-30, --fs-34, --fs-36, --fs-42, --fs-48, --fs-52
--fs-60, --fs-160
```

**エージェントがすべきこと:**
- 存在しないトークン → 最も近い既存トークンに置き換える
- 例: `var(--ink-40)` → `var(--ink-34)` または `var(--ink-50)`（文脈で判断）
- 例: `var(--sp-28)` → **SOFT 違反として報告**（後述。スペーシングの丸めは自動修正不可）

**例外:** `--soft-black` は `var(--ink-100)` のエイリアスとして slides.css に定義済み。使用は許容されるが、新規追加時は `var(--ink-100)` を推奨する。

**エージェントがしてはならないこと:**
- `var(--sp-28)` を `var(--sp-24)` や `var(--sp-32)` に自動的に丸めること（SOFT 違反に該当。レポートのみ行う）

---

### H06: ハードコード Hex カラーのうち正確なトークン等価があるもの

**定義:** インラインスタイルまたは `<style>` ブロック内に、DS トークンと**完全に一致する**ハードコード値が使用されている。

**完全一致マッピング:**

| ハードコード値 | トークン |
|---|---|
| `#1a1a1a` | `var(--ink-100)` |
| `#ffffff` / `#fff` | `var(--surface-white)` |
| `#f7f7f7` | `var(--surface-mist)` |
| `#5a4970` | `var(--accent-purple)` |
| `#9e3434` | `var(--accent-red)` |

**なぜ HARD か:** 同一の値をトークン参照に置き換えるだけなので、表示は一切変わらない。

**エージェントがすべきこと:**
```css
/* NG */
color: #1a1a1a;
background: #ffffff;

/* OK */
color: var(--ink-100);
background: var(--surface-white);
```

**エージェントがしてはならないこと:**
- `#555555`, `#666`, `#888`, `#999`, `#bbb` 等、**トークンと完全一致しない**グレー値を自動変換すること。これらは SOFT 違反（後述）
- SVG の `fill` / `stroke` に使われている色を、文脈を確認せずに変換すること（図形の配色はデザイン意図が強い）

**注意: `color: #fff` と `background: #fff` は区別する。**
- `color: #fff` → `color: var(--surface-white)` は正しい
- `background: #fff` → `background: var(--surface-white)` は正しい
- ただし、`background: #ffffff` が `.slide-container` の背景として使われている場合は `var(--slide-bg-paper)` がより適切な場合がある

## SOFT 推奨事項（レポートのみ、自動修正禁止）

以下の項目は**修正するとレイアウトが変わる可能性がある**ため、エージェントは検出・レポートのみを行い、修正は人間またはビジュアルレビュー付きの別プロセスに委ねる。

### S01: 標準スケール外のスペーシング値

**定義:** `margin`, `padding`, `gap` に標準スペーシングスケール（4, 8, 12, 16, 20, 24, 32, 48）外のピクセル値が使用されている。

**なぜ SOFT か:** 前回の大規模修正で最も被害が大きかったカテゴリ。非標準値の多くは意図的な微調整であり、最寄りの標準値への丸めが累積ピクセルずれを生む。

**対象となる値の例:**
```css
/* レポート対象 */
padding: 72px 80px 68px;    /* slide_01: タイトルスライド固有の余白 */
gap: 6px;                    /* アイコンとテキストの微小間隔 */
gap: 1px;                    /* グリッドセパレータ */
padding: 14px;               /* 標準スケール外 */
margin-bottom: 10px;         /* 標準スケール外 */
gap: 14px;                   /* 標準スケール外 */
```

**レポート不要の例外:**
```css
/* 以下はスペーシングスケールの対象外であり、レポート不要 */
--stage-top: 184px;          /* ステージ座標（レイアウトジオメトリ） */
--stage-top: 188px;          /* ステージ座標 */
width: 260px;                /* 図形のサイズ */
height: 160px;               /* 図形のサイズ */
left: 48px;                  /* 図形の絶対座標 */
top: 24px;                   /* 図形の絶対座標 */
border-radius: 999px;        /* 完全な円 */
```

**エージェントがすべきこと:**
- 非標準スペーシング値を検出してレポートに記載する
- 最寄りの標準値と差分（何 px ずれるか）を付記する
- そのスライド内での使用文脈（何のための余白か）を記載する

**エージェントがしてはならないこと:**
- 自動的に最寄りの標準トークンに丸めること
- `gap: 6px` を `gap: var(--sp-8)` に変更すること
- `padding: 72px` を `padding: var(--sp-48)` に変更すること

---

### S02: margin vs gap の使用パターン

**定義:** 子要素に `margin-bottom` や `margin-top` でスペーシングを管理しており、親の `gap` に集約できる可能性がある。

**なぜ SOFT か:** DS のガバナンスルール 07（「スペーシングは親の gap に固定する。子の margin は使わない」）に基づくが、margin から gap への変換はレイアウトセマンティクスを変える。

- `margin` は折りたたみ（margin collapse）が発生する
- `gap` は折りたたみが発生しない
- 不均一な間隔（要素 A-B 間は 8px、B-C 間は 16px）を `gap` だけでは表現できない
- レガシースライドの多くは `margin-bottom` で縦リズムを管理しており、一括変換すると間隔が変わる

**エージェントがすべきこと:**
- margin でスペーシングを管理している箇所を検出してレポートに記載する
- gap への変換が可能かどうかの判断材料（均一か不均一か）を付記する

**エージェントがしてはならないこと:**
- `margin-bottom: var(--sp-16)` を削除して親に `gap: var(--sp-16)` を追加すること
- margin と gap が混在している親要素で片方を削除すること

---

### S03: 装飾ボーダーと構造ボーダーの区別

**定義:** `border-top`, `border-left`, `border-bottom` 等の片辺ボーダーが使用されている。

**なぜ SOFT か:** DS のガバナンスルール 10 は「装飾 border-top / border-left の禁止」を定めているが、以下の例外がある。

**除外対象（修正不要）:**
```css
/* 構造ボーダー: 情報の区切りを示す */
border-bottom: 1px solid var(--ink-08);   /* .ds-row のセパレータ */
border-top: 1px solid var(--ink-08);      /* .ds-summary の区切り線 */
border-bottom: 1px solid var(--ink-04);   /* アジェンダ行のセパレータ */

/* カードの全周ボーダー */
border: 1px solid var(--ink-08);          /* .ds-card の標準ボーダー */
border: 2px solid var(--ink-100);         /* .ds-card--strong */
```

**レポート対象:**
```css
/* 装飾ボーダー: 片辺を太線・色付きで強調している */
border-top: 4px solid #000;              /* カード上部の強調線 */
border-left: 3px solid var(--accent-purple);  /* パネル左辺のアクセント */
```

**エージェントがすべきこと:**
- 片辺ボーダーを検出し、構造的（セパレータ）か装飾的（強調）かを判断してレポートする
- 装飾的と判断した場合、代替案（塗り、タイポ、バッジ、全周の細い枠）を提案する

**エージェントがしてはならないこと:**
- `.ds-row` や `.ds-summary` のセパレータボーダーを削除すること
- `border-bottom: 1px solid var(--ink-08)` を自動的に「装飾」と判定して削除すること
- 全周ボーダー `border: 1px solid var(--ink-08)` を片辺ボーダーと混同すること

### S04: line-height の値

**定義:** `line-height` の値が DS で定義されたパターン（1.0, 1.1, 1.14, 1.2, 1.25, 1.4, 1.45, 1.5, 1.55, 1.6, 1.65, 1.7, 1.72, 1.75, 1.8）と異なる。

**なぜ SOFT か:** line-height はテキストの折り返し位置とコンテンツ領域の高さに直結する。1280x720px の固定キャンバスでは、line-height が 0.05 変わるだけで 2 行テキストの高さが数ピクセル変わり、下のコンテンツが押し出される可能性がある。

**エージェントがすべきこと:**
- 非標準の line-height 値を検出してレポートする

**エージェントがしてはならないこと:**
- `line-height: 1.35` を `line-height: 1.4` に自動変更すること

---

### S05: 図形・ツリーレイアウトのハードコードピクセル値

**定義:** 絶対配置（`position: absolute`）された図形、SVG 座標、CSS で描画されたツリー/フローチャートのノード位置に使用されているハードコードピクセル値。

**なぜ SOFT か:** これらの値は幾何学的な配置であり、スペーシングトークンとは本質的に異なる。ノードの位置を 4px 移動しただけで、コネクタ線との接続が崩れる。

**対象例:**
```css
/* これらは全てレポート不要（図形座標） */
left: 48px;
top: 24px;
width: 94px;
height: 94px;
cx="150" cy="48" r="10"    /* SVG 座標 */
```

```css
/* これは --stage-top であり、レイアウトジオメトリ（レポート不要） */
--stage-top: 184px;
```

**エージェントがすべきこと:**
- 図形座標のハードコードピクセル値は**無視する**（レポートにも含めない）
- `--stage-top` のような DS レイアウトトークンの値は**無視する**

**エージェントがしてはならないこと:**
- SVG の座標値を「ハードコードピクセル」として報告すること
- `--stage-top: 184px` を `--stage-top: var(--sp-48)` などに変換しようとすること
- CSS で描画されたフローチャートのコネクタ線の位置を変更すること

---

### S06: 標準スケール外の padding 値

**定義:** 要素の padding にスペーシングスケール外の値が使用されている。

**なぜ SOFT か:** S01 と同じ理由。特にカードやパネルの padding は視覚的な「呼吸」を決定しており、数ピクセルの変更がコンテンツの収まり方を変える。

**エージェントがすべきこと:**
- スケール外の padding 値を検出してレポートする
- `var(--pad-card-sm)`, `var(--pad-card-md)`, `var(--pad-card-lg)` のいずれかで置き換え可能かを付記する

**エージェントがしてはならないこと:**
- `padding: 22px 24px` を `padding: var(--sp-24)` に自動変換すること
- `padding: 14px` を `padding: var(--sp-12)` や `padding: var(--sp-16)` に自動変換すること

---

### S07: 標準スケール外の font-size

**定義:** `font-size` に DS のフォントサイズスケール外の値が使用されている。

**なぜ SOFT か:** フォントサイズの変更はテキストの折り返し位置と行数を変え、固定キャンバス内のレイアウトに影響する。

**DS フォントサイズスケール:**
```
11, 12, 13, 14, 15, 16, 17, 18, 22, 24, 30, 34, 36, 42, 48, 52, 60, 160
```

**エージェントがすべきこと:**
- スケール外の font-size を検出してレポートする（例: `font-size: 20px`, `font-size: 28px`）

**エージェントがしてはならないこと:**
- `font-size: 20px` を `font-size: var(--fs-18)` や `font-size: var(--fs-22)` に自動変換すること

---

### S08: ハードコードグレー値（トークンと不完全一致）

**定義:** `#555`, `#555555`, `#666`, `#888`, `#999`, `#bbb` など、DS の ink スケールと完全一致しないグレー値が使用されている。

**なぜ SOFT か:** これらの値は ink スケールのいずれかに「近い」が「完全一致しない」。どの ink トークンに対応させるかはデザイン判断が必要。

**参考マッピング（自動修正不可、レポート時の参考情報として付記）:**

| ハードコード値 | 近似する opacity | 候補トークン |
|---|---|---|
| `#555` / `#555555` | ~0.67 | `--ink-68` |
| `#666` / `#666666` | ~0.60 | `--ink-68` or `--ink-50` |
| `#888` / `#888888` | ~0.47 | `--ink-50` or `--ink-34` |
| `#999` / `#999999` | ~0.40 | `--ink-34` |
| `#bbb` / `#bbbbbb` | ~0.27 | `--ink-34` or `--ink-18` |

**エージェントがすべきこと:**
- ハードコードグレー値を検出し、近似トークンの候補を付記してレポートする

**エージェントがしてはならないこと:**
- 上記マッピングに基づいて自動変換すること

---

### S09: インラインスタイルの CSS クラス化

**定義:** 複数のスライドで繰り返し使用されている同一のインラインスタイルパターンが、CSS クラスとして定義されていない。

**なぜ SOFT か:** インラインスタイルを CSS クラスに抽出する行為自体は安全だが、スライド固有の微調整（`--stage-top`, 個別の gap 値等）がインラインスタイルに含まれていることが多く、単純な抽出ではスライド固有のチューニングが失われる。

**エージェントがすべきこと:**
- 3 枚以上のスライドで同一パターンが使われているインラインスタイルを検出し、クラス化の候補としてレポートする
- ガバナンスルール 09（「3 枚以上で同じパターン → ds-* クラスに昇格」）を参照する

**エージェントがしてはならないこと:**
- レポートなしにインラインスタイルを CSS クラスに置き換えること
- スライド固有のカスタムプロパティ（`--stage-top`, `--stage-gap`）をクラス化しようとすること

---

### S10: スライド固有カラー値（意図的な非トークンカラー）

**定義:** `#626c78`, `#5f6875` など、DS トークンに存在しないカラー値がスライドのコンテンツ要素（バー、バッジ等）に使用されている。

**なぜ SOFT か:** 一部のスライド（slide_02 等）では、組織図や棒グラフの要素にブランドカラーや識別カラーとして意図的にカスタムカラーが使用されている。これらを ink スケールに丸めるとコンテンツの識別性が失われる。

**エージェントがすべきこと:**
- DS トークンに存在しないカラー値を検出してレポートする
- 使用文脈（図形の塗り、テキスト色、背景色）を付記する

**エージェントがしてはならないこと:**
- コンテンツ固有のカラー値を ink スケールに自動変換すること
